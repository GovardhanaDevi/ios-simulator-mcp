SLIDE 1 · The 60-second pitch

What is MCP?

Model Context Protocol is an open standard from Anthropic for connecting AI assistants to external tools, data, and systems. Think of it as USB-C for LLMs: one cable, many devices.

Wire format

JSON-RPC 2.0

Every message is a structured JSON request/response. Same shape whether the transport is stdio, HTTP, or SSE.

Three primitives

Tools · Resources · Prompts

Tools = actions the model can call. Resources = read-only data. Prompts = reusable templates.

Transport-agnostic

stdio (this MCP) · HTTP · SSE

Our server speaks stdio: the host process pipes JSON in and out via stdin/stdout.

Why it matters: before MCP, every AI app reinvented its own tool-calling layer. Now Claude Desktop, Cursor, Claude Code, Cowork, and others can all talk to the same server we ship.

SLIDE 2 · The big picture

How a request flows

simctl path

Anything Apple ships with Xcode: boot, screenshot, install IPA, GPS, video record. Fast. Sim-only or via xcdevice.

WDA path

Anything that requires touching pixels: tap, swipe, type, hardware buttons, UI tree. Real device too. Slower to start (~30 s).

SLIDE 3 · Protocol anatomy

Three things an MCP server exposes

1. Tools

Things the model can do

Side-effecting actions: tap, screenshot, boot_simulator. Each has a JSON Schema for inputs.

← Our MCP ships 27 of these.

2. Resources

Things the model can read

Read-only data identified by URI: log files, project source, config. Not used by our MCP yet — could expose sim system logs as a resource later.

3. Prompts

Reusable templates

Pre-built prompts the user can pick from a menu. e.g. "QA my app: launch, take screenshots of every tab." Also not used yet.

The lifecycle

initialize — client and server handshake on protocol version & capabilities
tools/list — server returns the 27 Tool objects with name, description, inputSchema
Model decides to call a tool. Client sends tools/call with name + args
Server runs the handler, returns CallTool.Result with content blocks (text, image, etc.)
Client feeds the result back into the model's context. Loop continues.

SLIDE 4 · The wire format

What actually flows over stdin/stdout

Every message is a JSON-RPC envelope. Newline-delimited. stderr is reserved for logs (which is why our log() helper writes there).

Client → server: tools/call

{
  "jsonrpc": "2.0",
  "id": 42,
  "method": "tools/call",
  "params": {
    "name": "tap",
    "arguments": {
      "x": 196,
      "y": 412
    }
  }
}

Server → client: result

{
  "jsonrpc": "2.0",
  "id": 42,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "Tapped at (196, 412)"
      }
    ],
    "isError": false
  }
}
The Swift MCP SDK hides the JSON-RPC plumbing. We register handlers and return strongly-typed CallTool.Result values; the SDK serializes them.

SLIDE 5 · How we describe a tool to the model

The schema is the contract

The inputSchema is the only thing the model sees when it decides whether and how to call a tool. Bad schema = bad calls. Each of our 27 tools has one. 

Tool(
    name: "tap",
    description: "Tap at x,y coordinates on the iOS Simulator screen.",
    inputSchema: .object([
        "type": .string("object"),
        "properties": .object([
            "x": .object(["type": .string("number"),
                              "description": .string("X coordinate in points")]),
            "y": .object(["type": .string("number"),
                              "description": .string("Y coordinate in points")]),
            "udid": .object(["type": .string("string"),
                                 "description": .string("Simulator UDID (optional)")]),
        ]),
        "required": .array([.string("x"), .string("y")]),
    ])
)
Lesson learned: describe units ("in points"), defaults ("uses booted sim if omitted"), and enums (e.g. button names: home/lock/siri). Vague schemas make Claude guess.

SLIDE 6 · How a screenshot actually happens

The simctl path, end-to-end

Claude calls screenshot with optional output_path.
ToolDispatcher.dispatch routes to SimctlTools.screenshot().
Handler asks SimulatorManager for the booted UDID — cached after first lookup.
Spawn /usr/bin/xcrun simctl io <udid> screenshot --type png /tmp/shot.png via async Process.
Read PNG bytes from disk.
Return CallTool.Result(content: [.image(data:, mimeType: "image/png")]).
Claude receives the image inline and can see it in the next turn.

No private APIs. No accessibility hacks. Just a CLI Apple ships with Xcode, wrapped in a Process with stdout/stderr piped, awaited from Swift concurrency, and re-emitted as MCP content.

SLIDE 7 · How a tap actually happens

The WDA path, end-to-end

User: "tap the Sign In button at 196, 412"
WDATools.tap() calls WDAManager.tap(x:y:).
Manager lazily creates a WDA session on http://localhost:8100/session if needed.
POSTs a W3C-style action to /session/<id>/actions:
WebDriverAgent (running inside the sim as an XCTest) translates that JSON into XCUICoordinate.tap().
iOS dispatches a real touch event. UIKit reacts as if a finger pressed the screen.

The actions payload we POST

{
  "actions": [{
    "type": "pointer",
    "id": "finger1",
    "parameters": { "pointerType": "touch" },
    "actions": [
      { "type": "pointerMove",
        "duration": 0,
        "x": 196, "y": 412 },
      { "type": "pointerDown", "button": 0 },
      { "type": "pause",       "duration": 100 },
      { "type": "pointerUp",   "button": 0 }
    ]
  }]
}

Same W3C WebDriver shape Selenium and Appium use — the standard's been around long enough that we get to inherit it for free.

SLIDE 8 · The most magical/scariest piece

How WDAManager starts WebDriverAgent

WDA isn't a service we install — it's an XCTest target that runs inside the simulator. We launch it with xcodebuild test and watch its log output to know when it's ready. 

// 1. Spawn xcodebuild test in background
let args = [
  "test",
  "-project", wdaProjectPath,
  "-scheme", "WebDriverAgentRunner",
  "-destination", "id=\(udid)",
  "USE_PORT=8100",
]
let proc = try launchBackground(
  "/usr/bin/xcodebuild", args) { line in
    log("[WDA] \(line)")
  }

// 2. Poll /status until ready (timeout 120s)
while Date() < deadline {
  if let s = try? await getStatus(),
     s.ready == true { return }
  try await Task.sleep(nanoseconds: delay)
  delay = min(delay * 2, 8_000_000_000)
}

What this gives us

Public API only. WDA uses XCTest, which Apple sanctions. No private headers, no jailbreak.

Simulator and device. Same code path. Real device just needs iproxy 8100 8100 for USB port-forward.

Battle-tested. Appium and Sauce Labs run on top of WDA. We're not inventing UI automation; we're putting an MCP face on it.

Cost

~30s cold-start while xcodebuild compiles and launches the runner. Subsequent calls reuse the same WDA session.

Concurrency

Why actor, not class

Swift 6 strict concurrency. The cached UDID and the recording Process are mutable shared state. Actors give us isolation for free; no manual locks.

stdout discipline

Logs go to stderr

log() writes to FileHandle.standardError because stdout is the protocol channel. A stray print() would corrupt the JSON-RPC stream.

SLIDE 12 · Anticipated questions

Q&A prep

Why Swift instead of TypeScript or Python like most MCP servers?

Three reasons. (1) We're already in the Apple ecosystem — xcrun, xcodebuild, and Foundation are first-class. (2) Process + actors give us robust concurrency without a runtime. (3) A native binary ships as a single executable; no node_modules, no venv.

Why not just call WDA directly from Claude using HTTP?

Claude can't speak HTTP to arbitrary localhost services from a chat. MCP is the bridge. Also, the MCP wraps WDA and simctl behind one consistent tool surface, with retry/session/cache logic the model shouldn't have to manage.

What happens if WDA crashes mid-session?

WDAManager.session() validates the cached sessionId with a /status ping. If it 404s, we drop it and create a new one on the next call. If xcodebuild exits, the next tool call will fail with a clear "WDA is not running" error — the model can re-run start_wda.

Real device support?

Same code path. WDA is built for the device target instead of the sim. The user runs iproxy 8100 8100 from libimobiledevice to forward USB to localhost, and start_wda sees it on port 8100 just like a sim.

Security — is Claude tapping arbitrary points scary?

Yes, and that's by design. The MCP runs locally as the user, so it has the same scope the user has. The host (Claude Code, Cowork) gates each tool call with user approval the first time. We don't expose anything Claude couldn't get by SSHing into the same machine.

Performance? How fast is a tap?

simctl tools: ~50–200ms (one fork/exec). WDA tools after warm-up: ~100–300ms per HTTP round-trip. The expensive thing is start_wda at ~30s cold (xcodebuild compiles WebDriverAgentRunner). We do it once per session.

Why not use idb like the Facebook MCP?

idb requires a Python companion process, an Objective-C bridge, and pulls in fb-idb daemons. It also relies on private frameworks for some operations. WDA is sanctioned (XCTest is a public Apple framework), already maintained by Appium, and works for sim + real device. Cleaner trade.

Could we add new tools easily?

Yes — add a Tool entry in ToolDefinitions.swift, a case in ToolDispatcher, and a handler in SimctlTools or WDATools. ~20 lines per tool. Good candidates: biometric simulation (simctl ui), accessibility settings, push-notification injection (simctl push).

Does Claude "see" the screen?

Only when we call screenshot. The image is returned as MCP image content and lands in the model's next turn as visual context. ui_describe_all is the cheaper text alternative — XML accessibility tree, no vision tokens.
